# Лабораторна робота №5

**Тема:** Побудова навігації у React Native із використанням бібліотеки Expo Router

**Мета:** Ознайомлення з концепцією file-based маршрутизації в мобільних застосунках

---

## Інструкція запуску

### Вимоги

- Node.js версії 18 або вище
- npm версії 9 або вище
- Встановлений застосунок Expo Go на мобільному пристрої або налаштований Android/iOS емулятор

### Кроки запуску

1. Клонувати репозиторій:

```bash
git clone https://github.com/MobileLabsRN2026/lab5
cd lab5
```

2. Встановити залежності:

```bash
npm install
```

3. Встановити необхідні пакети Expo:

```bash
npx expo install expo-router expo-linking expo-constants expo-status-bar react-native-safe-area-context react-native-screens @react-native-async-storage/async-storage
```

4. Переконатися, що у файлі `package.json` вказана коректна точка входу:

```json
{
  "main": "expo-router/entry"
}
```

5. Запустити проєкт з очисткою кешу:

```bash
npx expo start --clear
```

6. Відсканувати QR-код у застосунку Expo Go або натиснути `a` для запуску на Android-емуляторі, `i` — для iOS.

## Опис реалізованого функціоналу

### File-based маршрутизація (Expo Router)

Навігація побудована на основі файлової структури директорії `/app`. Кожен файл у цій директорії автоматично стає окремим маршрутом без необхідності ручної реєстрації екранів. Назви файлів безпосередньо відповідають URL-адресам маршрутів.

### Групи маршрутів

Проєкт містить дві групи маршрутів:

- `(auth)` — публічна група, доступна без авторизації. Містить екрани входу та реєстрації. Назва групи не відображається в URL, тобто маршрути доступні як `/login` та `/register`.
- `(app)` — захищена група, доступна лише авторизованим користувачам. Містить каталог товарів та екран деталей.

### Контекст авторизації

Реалізовано глобальний `AuthContext` з використанням React Context API. Контекст зберігає:

- `isAuthenticated` — булевий стан авторизації
- `user` — об'єкт з даними поточного користувача (ім'я, email)
- `isLoading` — стан завантаження під час читання збереженої сесії

Доступні методи: `login(email, password)`, `register(email, password, name)`, `logout()`.

### Збереження даних користувачів

Дані зареєстрованих користувачів зберігаються локально за допомогою `AsyncStorage`. При реєстрації перевіряється унікальність email-адреси. При повторному запуску застосунку активна сесія відновлюється автоматично без необхідності повторного входу.

### Захищені маршрути

У файлі `app/(app)/_layout.jsx` реалізована перевірка стану авторизації. Якщо користувач не авторизований (`isAuthenticated === false`), відбувається автоматичне перенаправлення на екран входу за допомогою компонента `<Redirect href="/login" />`.

### Екран входу

Містить поля для введення email та пароля, кнопку входу та посилання на екран реєстрації. При введенні некоректних даних відображається повідомлення про помилку через `Alert`. Після успішного входу користувач перенаправляється до каталогу товарів.

### Екран реєстрації

Містить поля для введення імені, email, пароля та підтвердження пароля. Реалізована перевірка на заповненість усіх полів та збіг паролів. Після успішної реєстрації користувач автоматично авторизується і переходить до каталогу.

### Каталог товарів

Головний захищений екран відображає список із 6 товарів у вигляді сітки з двох колонок за допомогою компонента `FlatList`. Кожна картка товару містить зображення, назву та ціну. Для переходу на екран деталей використовується компонент `<Link asChild>`, який передає поведінку навігації дочірньому елементу. У шапці відображається ім'я авторизованого користувача та кнопка виходу.

### Деталі товару (динамічний маршрут)

Екран `app/(app)/details/[id].jsx` реалізує динамічний маршрут. Параметр `id` отримується з URL за допомогою хука `useLocalSearchParams()`. На основі отриманого ідентифікатора з масиву даних вибирається відповідний товар і відображається повна інформація: зображення, назва, ціна та детальний опис. Якщо товар не знайдено, відображається відповідне повідомлення.

### Обробка неіснуючих маршрутів

Файл `app/+not-found.jsx` перехоплює всі звернення до неіснуючих маршрутів. На екрані відображається повідомлення "Екран не знайдено" та кнопка повернення на головну сторінку.

## Висновки

### Відповіді на контрольні запитання

**1. Яким чином за допомогою Expo Router реалізується перенаправлення неавторизованого користувача?**

Перенаправлення реалізується безпосередньо у файлі `_layout.jsx` захищеної групи маршрутів. При рендері layout отримує поточне значення `isAuthenticated` з глобального контексту. Якщо значення дорівнює `false`, компонент повертає `<Redirect href="/login" />` замість навігатора. Expo Router обробляє цей компонент як негайне перенаправлення ще до відображення будь-якого вмісту захищеної групи. Цей підхід є декларативним і автоматично спрацьовує при будь-якій спробі доступу до захищених маршрутів.

**2. У чому полягає різниця між використанням компонента `<Link>` та методом `router.push()`?**

`<Link>` є декларативним JSX-компонентом, який використовується безпосередньо у розмітці. Він рендериться як елемент, що підтримує нативну поведінку навігації, у тому числі prefetching на вебі. Пропс `asChild` дозволяє передати поведінку навігації дочірньому компоненту, наприклад `TouchableOpacity`, зберігаючи власні стилі. `router.push()` є програмним методом навігації, який викликається всередині функцій і обробників подій, наприклад після успішної валідації форми або завершення асинхронної операції. Обидва підходи виконують одну задачу, але `<Link>` підходить для статичних переходів у розмітці, а `router.push()` — для умовної або відкладеної навігації в логіці компонента.

**3. Як працюють динамічні маршрути в Expo Router і як отримати передані параметри?**

Динамічний маршрут створюється шляхом іменування файлу з параметром у квадратних дужках, наприклад `[id].jsx`. Expo Router автоматично розпізнає такий файл як маршрут із змінним сегментом URL. При переході на адресу `/details/3` Expo Router зіставляє її з файлом `details/[id].jsx` і передає значення `3` як параметр `id`. Для отримання цього параметру всередині компонента використовується хук `useLocalSearchParams()`, який повертає об'єкт з усіма параметрами поточного маршруту.

**4. Чому стан авторизації доцільно зберігати у глобальному контексті, а не в локальному стані компонента?**

Локальний стан компонента існує лише в межах його lifecycle і недоступний іншим компонентам без явної передачі через props. Стан авторизації потрібен одночасно в багатьох частинах застосунку: у layout для перевірки доступу, у каталозі для відображення імені користувача, у будь-якому екрані для виклику `logout()`. Зберігання цього стану в React Context забезпечує єдине джерело істини для всього дерева компонентів. Будь-який компонент може отримати доступ до стану через хук `useAuth()` без prop drilling. При зміні стану всі підписані компоненти автоматично отримують актуальні дані і перерендеряться.

**5. Для чого використовуються групи маршрутів і як вони впливають на URL-адресу?**

Групи маршрутів створюються шляхом іменування директорії в дужках, наприклад `(auth)` або `(app)`. Вони слугують виключно для логічної організації файлів проєкту та застосування спільного layout до певного набору екранів. Принциповою особливістю є те, що назва групи повністю ігнорується при формуванні URL-адреси. Файл `app/(auth)/login.jsx` доступний за адресою `/login`, а не `/(auth)/login`. Це дозволяє розділяти екрани на логічні групи зі спільними layout та поведінкою (наприклад, захист авторизацією) без зміни структури URL.
